# Fast Response Default

## Зачем

Когда запрос не требует высокой точности, дорогих проверок и доказательной базы, отвечать быстрее и уже, не начиная с широкой разведки по файлам, логам и БД.

## Правило по умолчанию

1. Сначала дать короткий оперативный ответ по верхнему уровню.
2. Не читать файлы, не идти в БД и не запускать широкие поиски, если можно ответить разумно и безопасно без них.
3. Углубляться только если:
- пользователь явно просит точность, проверку, статистику, evidence, ссылки на файлы или актуальное состояние системы
- без проверки есть высокий риск ошибки
- вопрос касается текущего состояния кода, БД, очередей, логов, конфигов или недавних изменений
4. Если нужна проверка, начинать с самого дешевого узкого шага, а не с широкой разведки.

## Предпочтительный стиль ответа

- Сначала 1 короткий ответ по сути.
- Потом, если уместно, 1 строка с оговоркой уровня уверенности: `быстро/предварительно` или `проверено`.
- Детали, доказательства и расширенные раскопки только по запросу.

## Как выбирать глубину

### Быстрый режим

Использовать по умолчанию для:
- общих вопросов
- операционных вопросов без требования точных цифр
- объяснений `почему медленно`, `что это`, `как лучше`
- оценок и гипотез, где достаточно явно пометить неопределенность

### Точный режим

Использовать, если пользователь просит:
- точные числа
- статистику
- состояние пайплайна/БД/очереди
- проверку файлов/логов/кода
- подтверждение, что что-то действительно произошло

## Практические ограничения

- Не делать параллельную разведку по нескольким источникам без необходимости.
- Не читать большие файлы целиком, если хватает узкого диапазона строк или `rg`.
- Не ходить в БД, если ответ можно получить из уже известных данных или локального лога.
- Если вопрос допускает два слоя ответа, сначала отдавать быстрый слой и предлагать добор точности отдельным запросом.

## Формулировки

Если отвечаю без глубокой проверки, можно кратко помечать:
- `предварительно`
- `по верхнему уровню`
- `без проверки БД`

Если перехожу к более дорогой проверке, сначала коротко предупреждать:
- `Если нужна точная цифра, сейчас быстро сниму её из БД.`

## Приоритет

Это правило действует по умолчанию в прямом диалоге с пользователем, если нет более строгого запроса на точность или явных инструкций на обязательную проверку.
