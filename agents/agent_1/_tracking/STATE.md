## Демо-контур (schema demo)

Снимок от: 2026-06-30 (структура подтверждена 2026-07-14)

- Подтверждённая структура `demo` (localhost » mvp_db): `demo.raw_items` ~57 000,
  `demo.clean_items` ~50 084 (ссылка = `url`), `demo.doc_labels` ~355 576
  (`impact ∈ neutral/positive/negative`; негатива ~17, позитива ~56),
  `demo.kr` = 5 KR (id 2–6). Обоснование метки — в `doc_labels.raw_json.why_for_goal`.
- Изолированный демо-стенд живёт в схеме `demo` базы `mvp_db` и в директории `agents/agent_1/demo/`.
- В `demo.raw_items` лежит замороженное июньское сырьё Parsers360 за диапазон `2026-06-01..2026-06-30`.
- В `demo.clean_items` лежит тот же корпус после exact-дедупа по `content_hash` и near-dup по MinHash-LSH (`num_perm=64`, `b=16`, `r=4`, threshold `0.7`), включая пометки `is_duplicate` и `dup_of`.
- В `demo.kr` сохраняются пользовательские запросы и enrichment от `agent_2`.
- В `demo.doc_labels` сохраняются результаты синхронной трёхшаговой разметки `agent_1` для демо-прогонов.
- `demo/demo_ingest.py` поднимает схему `demo`, загружает июньский диапазон, строит frozen corpus и печатает сводку по объёму, источникам и дням.
- `demo/demo_run.py` выполняет линейный демо-прогон `запрос -> enrichment -> filter -> labeling`, пишет человекочитаемый trace в stdout и дублирует его в `demo/run_trace.log`.
- Это отдельный демонстрационный контур. Он не использует `agent_1.processing_jobs`, не запускает боевые воркеры и не пишет в схему `agent_1`.

## Рабочие инструкции (Q&A по demo)

- Единственный источник данных — `mvp_db` схема `demo` (read-only через `AGENT_1_DB_DSN`).
  Таблицы: `demo.kr` (5 KR), `demo.doc_labels` (`impact`, `kr_id`, `clean_item_id`, `raw_json`),
  `demo.clean_items` (`url` = ссылка, `title`, `source`), `demo.raw_items`.
- KR всегда брать из `demo.kr.text`; их ровно 5 (id 2–6).
- На запрос примеров тональности (напр. «приведи пример негативной тональности»):
  взять `text` из `demo.kr` → в `demo.doc_labels` отобрать `impact='negative'`
  → сматчить по `kr_id` и `clean_item_id` → ссылку взять из `demo.clean_items.url`
  → обоснование из `raw_json->>'why_for_goal'` (+ цитата из `raw_json->'mentions'`).
- Формат по каждому примеру: **КР** / **Влияние** / **Обоснование** / **Ссылка**.
  По умолчанию ровно 3 примера; при повторном одинаковом запросе — новая подборка;
  если подходящих строк меньше 3 — отдать сколько есть и указать фактическое число.
- Стиль — формальный отчёт, ключевые части жирным. Не писать служебных фраз про
  fallback, пустоту боевой разметки, контуры БД или «в сохранённой разметке указано»;
  сразу отчёт и фактическая причина влияния.
- Канонический SQL и вариации (positive / по конкретному KR / счётчики) — в `DEMO_QA.md`.
