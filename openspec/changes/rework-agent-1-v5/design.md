## Context

Система переходит на 6-агентный пайплайн v5
(`docs/architecture/pipeline_v5_context.md`,
`docs/architecture/schema_v5.dot`). Агент 1 сегодня — монолит: сбор
(Parsers360 → `raw_items`), препроцессинг (`clean_items`) и шесть LLM-шагов
на документ. Main-спеки в `openspec/specs/` фиксируют это состояние «как
есть» (снимок 2026-07-17, архив `document-agent-1-pipeline`).

Измеренная фактура по производительности:
- препроцессинг: 10 381 raw/час ≈ 345 мс/док; чистые вычисления ≈ 22 мс/док
  (MinHash 128 перм + шинглы word-5/char-17 на 4000 симв, 1 ядро) → ~94%
  времени — per-document round-trips в БД;
- LLM-разметка: медленность = недостаток конкурентности, не модели; плюс
  баг: document-level шаги (tonality, paid_news) вызываются на каждый КР с
  идентичным входом.

Корпус для приёмки: текущие ~57k raw-документов.

## Goals / Non-Goals

**Goals:**
- Агент 1 = только сбор → очистка → дедуп → эмбеддинги; ноль LLM-вызовов.
- Схема v5 для зоны Агента 1: `source`, `raw_posts`, `clean_posts`.
- ≤25 мс/док на ядро; ≥400k raw/час на 4 ядрах (без эмбеддингов).
- Идемпотентность: повторный прогон — 0 эмбеддинг-вызовов, 0 изменённых
  строк.
- Результат очистки/дедупа эквивалентен текущему (допуск ≤0.5% по множеству
  отбракованных).

**Non-Goals:**
- Агенты 2–6 (свои changes), партиционирование, agent_memory, дистилляция
  LLM, перекалибровка порогов дедупа, GLiNER (сначала оффлайн-проверка).

## Decisions

### D1. Отбраковка — строка в `clean_posts` со статусом, не лог-таблица
Каждый raw получает вердикт в `clean_posts` (`drop_reason IS NULL` = чистый;
`non_russian`, `junk:<категория>`, `duplicate` и т.д.). Альтернатива —
отдельная лог-таблица отбраковки — отклонена: статистика источников
(`agent_1-source-stats`) потребовала бы JOIN двух источников правды, а
инвариант «clean_posts = только чистое» всё равно уже нарушен полем
`is_duplicate`. Новый инвариант: **clean_posts = вердикт очистки по каждому
обработанному raw**. Отбракованные строки не получают `clean_content` и
`embedding`.

### D2. Junk-категории — таблица в БД
Regex-категории (текущие 17 + business-guard) переезжают из кода в таблицу:
пополнение без деплоя — прямое требование. Воркер кэширует активные правила
в память на время жизни батча/процесса. Альтернатива (конфиг в репо)
отклонена: любое изменение = деплой.

### D3. Пороги дедупа переносятся как есть
MinHash 128/32×4, шинглы word-5/char-17, Jaccard ≥0.7, levenshtein-пороги —
без изменений. Иначе критерий приёмки «эквивалентность ≤0.5%» не отличит
регрессию реализации от смены калибровки. Перекалибровка — отдельный change.

### D4. Вектора — путь A: Postgres/pgvector, dims=1024
Отдельная векторная БД отклонена: ни один шаг не делает ANN по большому
множеству (кластеризация батчевая, линкинг — брутфорс по сотням центроидов).
1024 (MRL-обрезка при генерации) влезает в HNSW без halfvec и ×3 дешевле по
памяти/CPU. HNSW (`vector_cosine_ops`) — только на `clean_posts.embedding`;
строится после bulk-загрузки. Эмбеддинг — расходный кэш: при нехватке места
обнуляется у холодных данных, текст не трогается.

### D5. Производительность: батчи + multiprocessing + смена MinHash-схемы
- Claim: `SELECT ... FOR UPDATE SKIP LOCKED LIMIT 100`, обработка пачки,
  запись одной транзакцией (bulk-вставка в `clean_posts` + один update
  статусов). Per-document транзакции запрещены.
- LSH-индекс в памяти воркера; БД — fallback-подтверждение между воркерами
  (текущая механика), не первичный поиск кандидатов.
- Multiprocessing по ядрам.
- cProfile до/после (100 документов, реальный корпус, не синтетика)
  прикладывается к PR.
- MinHash: «просто добавить numpy» на текущую формулу не работает.
  Профилирование (2026-07-20, локально) подтвердило, что >90% времени
  внутри `build_minhash_signature_from_shingles` — вложенный цикл
  `128 перестановок × N шинглов`. Но текущая схема — `(a*x + b) mod
  (2⁶¹−1)`, где произведение `a*x` не влезает в `uint64` (оба операнда
  до ~64 бит) → numpy вынужден считать в `object`-dtype, что даёт
  измеримо, но неубедительно (~+15-20%, не кратный выигрыш) вместо
  честной SIMD-векторизации.
  Решение: заменить схему хэш-миксинга на нативно `uint64`-совместимую
  — `(x XOR mask_i) * odd_multiplier_i` с естественным переполнением по
  2⁶⁴ (128 независимых масок/множителей вместо 128 пар `(a,b)`).
  Такая формула честно векторизуется broadcast'ом `(128, N)` +
  `min(axis=1)` без object-dtype. Безопасно менять: итоговый accept/
  reject near-dup MUST оставаться точным Jaccard по множествам шинглов
  (текущее поведение, код уже это делает — MinHash только для LSH-отбора
  кандидатов), т.е. смена схемы хэш-миксинга не трогает калибровку
  порогов (D3) — влияет только на recall кандидатов, не на вердикт.
  После этой замены новым узким местом становится конструирование
  шинглов (`blake2b` на каждый шингл) — MUST заменить на быстрый
  некриптографический хэш (MinHash не требует криптостойкости); иначе
  цель ≤25 мс/док на ядро не достигается только за счёт сигнатуры.
- Перед реализацией — перезамерить baseline (3.1) на реальных документах
  из продового корпуса, не на синтетическом тексте: состав/уникальность
  шинглов у формулировочного вебдискурса (шаблонный wire-текст,
  повторяющиеся обороты) может ощутимо отличаться от случайного текста.

### D6. LLM-код не удаляется, а выводится из состава
`label_kr_worker.py`, `extract_semantics_worker.py`, `label_prompts.py`,
`kr_enrichment_sync.py` исключаются из очереди/запуска Агента 1, но код
остаётся в репо до создания Агентов 2/4 — там он источник логики (и silver-
датасет для будущей дистилляции копится в существующей разметке). Удаление
физических файлов — при выносе в соответствующих changes.

### D7. Схема: новые таблицы + миграция, а не ALTER старых
`source`, `raw_posts`, `clean_posts` создаются заново по v5 (композитный PK
с `time_post`, LZ4-компрессия content-колонок, атомарные поля вместо
`raw_payload`-склейки). Данные мигрируются из `agent_1.raw_items` /
`clean_items` разовым скриптом. Старые таблицы остаются read-only до вывода
LLM-воркеров в Агенты 2/4 (они читают старую схему).

### D8. Кэшировать шинглы кандидатов в LSH-индексе, не пересчитывать
Находка при разборе D5 (2026-07-20): текущий `find_near_duplicate`
пересчитывает `build_shingles(candidate.normalized_text)` заново для
**каждого** кандидата из бэнда на **каждый** запрос — `NearDuplicateEntry`
кэширует `signature`/`band_keys`/`normalized_text`, но не сами шинглы.
Для часто копируемых новостей (одна и та же новость с десятков
источников — типичный случай для новостного корпуса) один и тот же
кандидат может пересчитываться многократно за прогон. Это отдельная от
MinHash-векторизации проблема — может доминировать в реальном near-dup
пути сильнее, чем сама конструкция сигнатуры, и не видна в бенчмарке
«построить сигнатуру для одного документа» (D5), только в бенчмарке
полного flow с реалистичным распределением дублей в корпусе.
Решение: `NearDuplicateEntry` MUST кэшировать `shingles: frozenset[int]`
наравне с `signature`/`band_keys` при построении/добавлении записи в
`NearDuplicateCache`, чтобы `find_near_duplicate` брал шинглы кандидата
из кэша, а не пересчитывал.

## Risks / Trade-offs

- [Эквивалентность дедупа при смене порядка обработки] → допуск ≤0.5% в
  критерии приёмки; прогон сравнения на полном корпусе 57k до merge.
- [In-memory LSH при multiprocessing: воркеры не видят кандидатов друг
  друга] → DB fallback-подтверждение сохраняется; допуск 0.5% покрывает
  расхождения на границах батчей.
- [Падение воркера посреди батча] → lease/retry по timeout (текущая
  механика), запись идемпотентна (upsert по `ID_raw_post`).
- [Rate limit API эмбеддера на bulk-прогоне] → батчи до лимита API,
  идемпотентность (`embedding IS NOT NULL` → пропуск) позволяет дожимать
  повторными запусками.
- [Миграция 57k+ строк со старой схемы] → разовый скрипт, прогон на копии;
  raw неприкосновенен, `clean_posts` пересобираем по определению.
- [Downstream-зависимость: Агент 3 пишет `clean_posts.ID_cluster`] →
  колонка закладывается сейчас (nullable FK), запись — вне скоупа.
- [Смена MinHash хэш-миксинга (D5) меняет recall LSH-кандидатов] →
  безопасно для порогов (accept/reject — точный Jaccard, не сигнатура),
  но recall кандидатов может сдвинуться на границе; покрывается тем же
  допуском ≤0.5% в критерии приёмки (3.1/6.1), отдельной перекалибровки
  не требует.

## Migration Plan

1. DDL новых таблиц + таблица junk-категорий (наполняется текущими 17
   категориями из кода) — без включения в пайплайн.
2. Миграция `raw_items` → `raw_posts`, `clean_items` → `clean_posts`.
3. Переключение ingestion на `raw_posts`; препроцессинг — на новый батчевый
   воркер с записью в `clean_posts`.
4. Отключение LLM-jobs из очереди Агента 1 (preprocess больше не ставит
   `label_kr`).
5. Bulk-эмбеддинг корпуса, затем построение HNSW.
6. Rollback: старые таблицы и воркеры не удаляются до приёмки — возврат =
   переключение очереди обратно.
