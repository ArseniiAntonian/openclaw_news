# Контекст: пайплайн v5 — что поменялось в системе и почему

Этот документ — вводная к изменениям системы. Он объясняет решения, чтобы читатель
(человек или агент) понимал, *почему* спеки такие, и не «чинил» намеренные вещи.

## 1. Новый пайплайн: 6 агентов вместо монолитной разметки

Было: Агент 1 делал всё — сбор, очистку, дедуп и **6 LLM-шагов на документ**
(relevance → impact → sber_paid_news → entity_tonality → entities → events),
результаты в `document_enrichments` / `document_kr_labels`.

Стало (оркестрирует Агент 6 / OpenClaw):

| Агент | Ответственность |
|---|---|
| 1 | сбор корпуса, RegEx-фильтр шума, дедуп (MinHash+levenshtein), **эмбеддинги** (text-embedding-3-small, dims=1024) |
| 2 | КР/цели: расшифровка, темы (типовые драйверы), ключевые слова, важность источников |
| 3 | кластеризация постов + мэтчи КР↔кластер |
| 4 | анализ: GLiNER-разметка каждой новости, извлечение драйверов, [action], приоритет |
| 5 | копирайтер: научный стиль, текст отчёта |
| 6 | оркестратор: очередь, чейнинг, ретраи |

Ключевое следствие для кода: **LLM-разметка уходит из Агента 1 полностью.**
Это не регресс, это перенос ответственности.

## 2. Центральное доменное решение: драйвер ≠ кластер

Драйвер — дискретное событие реального мира («ЦБ повысил ставку»), влияющее на КР.
Оно живёт **поперёк документов**: один драйвер = десятки публикаций.

- Идентичность драйвера = `normalized_key` вида `event|actor|object` (через словарь
  `entity_canonical`: «ЦБ/Центробанк/Банк России» → один канон). **НЕ через членство
  в кластере** — у кластеров плывущая идентичность между прогонами, у драйверов она
  должна быть стабильной.
- Кластер (Агент 3) — механизм дедупа/поиска кандидатов, не аналитическая сущность.
- Динамика драйвера во времени — **агрегат из `driver_mention`** (magnitude +
  created_at), отдельная таблица истории намеренно удалена как производная.
- GLiNER даёт **слоты** (спаны: акторы, объекты, номинализованные события), но НЕ
  связывание их в тройку — это принципиальное ограничение span-labeling. Сборка
  `(event|actor|object)` + direction/horizon — отдельный шаг (LLM или синтаксис),
  не GLiNER.

## 3. Схема БД: эволюция v1 → v5

- v1 (историческая): длинные цепочки (raw → expanded → enriched КР; analytics →
  insights → reports), ~31 таблица.
- v3 lean: сворачивание per-post свойств в колонки, 15 таблиц.
- v4: ещё −4 (parsers → колонка, enriched_KR → слит в KR, driver_type → varchar,
  magnitude_point → производная).
- **v5 (текущая, финальная)**: возврат стадийности — **каждый агент пишет в свои
  таблицы**, 13 штук:

`source, raw_posts, clean_posts` (Аг1) · `Key_results` (Аг2) ·
`cluster, kr_cluster_match` (Аг3) ·
`gliner_entities, entity_canonical, driver_instance, driver_mention` (Аг4) ·
`kr_report` (Аг4+5) · `processing_jobs, post_kr_processing` (Аг6)

Принципы, которые НЕ нарушать:
- `raw_posts` — только вывод парсера, поля атомарные (`title`, `url`, `content`,
  `time_post`, `lang` — отдельно). Raw неприкосновенен: никогда не удаляется и не
  мутируется очисткой.
- `clean_posts` — пересобираемый слой: можно сносить и перечищать, raw остаётся.
- `raw_posts.url` — пермалинк новости (для дашборда); `source.URL_source` — адрес
  издания. Не путать.
- Известное сознательное отступление от стадийности: `clean_posts.ID_cluster`
  пишет Агент 3.
- PK таблиц постов — композитный с `time_post` (задел под партиционирование;
  само партиционирование — отдельный change).

Исходники: `schema_v5.dot` (+ PNG), DBML истории — `schema_v3_lean.dbml`.

## 4. Вектора: путь A (без отдельной векторной БД)

Решение: **всё в Postgres/pgvector**, отдельная векторная БД (Qdrant и т.п.)
отклонена. Причина: ни один шаг пайплайна не делает ANN-поиск по большому
множеству — кластеризация батчевая (в память), линкинг драйверов и мэтч КР —
брутфорс по сотням центроидов.

- Размерность **1024** (`dimensions=1024` при генерации, MRL-обрезка) — влезает
  в HNSW без halfvec, ×3 дешевле по памяти/CPU.
- HNSW (`vector_cosine_ops`) — **только** на `clean_posts.embedding`. Центроиды
  кластеров/драйверов и эмбеддинг КР — без индексов (брутфорс быстрее на сотнях
  строк).
- HNSW строить **после** bulk-загрузки.
- Эмбеддинги — расходный кэш, текст — источник правды. При нехватке места обнулять
  embedding у холодных данных, текст не трогать (текст ≈ 2–3 КБ/док после LZ4,
  вектор+индекс ≈ 8 КБ/док).

## 5. Производительность: где на самом деле узкие места

Измерено, не предположено:

- Очистка: факт 10 381 raw/час ≈ **345 мс/док**; бенчмарк реальных вычислений
  (MinHash 128 перм + шинглы word-5/char-17 на 4000 симв) = **~22 мс/док** на
  1 ядре. → 94% времени — I/O (per-document round-trips в БД). Лечение: батчевый
  claim (`FOR UPDATE SKIP LOCKED LIMIT 100`), одна транзакция на пачку, in-memory
  LSH, векторизованный MinHash (6 мс/док), multiprocessing.
- LLM-разметка: медленность = недостаток конкурентности, не модели (350k вызовов
  × 3с: последовательно 12 суток, 100 воркеров — 3 часа). Найденный баг:
  document-level шаги (tonality, paid_news) зовутся по разу **на каждый КР** с
  идентичным входом — кэшировать по документу.
- Дистилляция маленькой LLM — отложена: чинит не то узкое место. Вернуться к ней,
  когда упрёмся в rate limits при высокой конкурентности; silver-датасет для неё
  уже копится в существующей разметке.

## 6. Отложено сознательно (не реализовывать без отдельного change)

1. Партиционирование по `time_post` (PK уже готов под него).
2. `agent_memory` — единая таблица памяти агентов (kind/scope/embedding/confidence);
   пока только копится статистика источников.
3. GLiNER в проде — сначала оффлайн-проверка скриптами `gliner_probe.py` /
   `gliner_inspect.py`: agreement с LLM-baseline и способность ловить
   номинализованные события. Если ловит — роль GLiNER: дешёвый фильтр воронки
   перед LLM-связыванием, не замена.
4. Кластерный уровень extraction (1 вызов на кластер вместо N на документы) —
   после того как Агент 3 стабилен.
