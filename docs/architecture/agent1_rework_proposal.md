# Change: Переделка Агента 1 — сбор, очистка, дедупликация, эмбеддинги

## Why

Агент 1 переводится на новую архитектуру пайплайна (v5): его зона ответственности
сужается до **сбор → очистка → дедуп → эмбеддинги**. Вся LLM-разметка (relevance,
impact, tonality, entities, events) из Агента 1 уходит: сущности переезжают в
Агент 4 (GLiNER + драйверный слой), оценка релевантности — на этап после
кластеризации.

Вторая причина — производительность. Текущая скорость препроцессинга
~10 400 raw/час (≈345 мс/документ) при измеренной стоимости реальных вычислений
~22 мс/документ (MinHash 128 перестановок + шинглование word-5/char-17 на 4000
символов, бенчмарк на 1 ядре). ~94% времени уходит не на вычисления —
предположительно на per-document round-trips в БД. Разрыв 16x до параллелизма,
~60x с учётом векторизации и нескольких ядер.

## What Changes

- Новая схема хранения: `source` → `raw_posts` → `clean_posts` (разделение сырого
  и очищенного, как в текущей agent_1, но с новыми полями).
- Эмбеддинги переезжают в Агент 1: `text-embedding-3-large`, `dimensions=1024`,
  колонка `clean_posts.embedding vector(1024)`.
- Удаление всех LLM-шагов из воркеров Агента 1 (**BREAKING**: таблицы
  `document_enrichments`, `document_kr_labels` больше не наполняются Агентом 1).
- Производительность: батчевый claim задач, in-memory LSH, векторизованный
  MinHash, multiprocessing.
- Наблюдаемость: агрегируемая статистика качества по источникам (задел под
  agent_memory).

## Non-Goals (вне скоупа этого change)

- Кластеризация (Агент 3), GLiNER-разметка и драйверный слой (Агент 4).
- Дистилляция/локальный инференс LLM.
- Партиционирование по `time_post` — проектируется, но включается отдельным
  change (см. решение ниже).
- agent_memory — только сбор статистики, без таблицы памяти.

## Требование 1: Схема данных

Агент 1 владеет тремя таблицами (Postgres + pgvector, `CREATE EXTENSION vector`):

### source
`ID_source PK, name_source, URL_source, type varchar, importance float, Is_active bool`
(бывший справочник source_type свёрнут в колонку `type`)

### raw_posts — только то, что отдал парсер, поля не склеиваются
`ID_raw_post PK, ID_source FK, parser varchar, title varchar, url varchar,
content text, time_post timestamptz, lang varchar, metadata jsonb,
collected_at timestamptz`

- `url` — пермалинк конкретной новости (обязателен для дашборда), не путать с
  `source.URL_source`.
- `content` и `clean_posts.clean_content` — компрессия LZ4:
  `ALTER TABLE ... ALTER COLUMN content SET COMPRESSION lz4;`

### clean_posts — результат очистки, 1:1 к прошедшим raw
`ID_clean_post PK, ID_raw_post FK, ID_canonical_post FK (self-ref, nullable),
ID_cluster FK (nullable, пишет Агент 3), clean_content text, is_duplicate bool,
dup_score float, embedding vector(1024), cleaned_at timestamptz`

#### Scenario: raw сохраняется всегда
- WHEN парсер вернул документ (включая мусор, дубли, не-русский)
- THEN строка в `raw_posts` создаётся безусловно; отбраковка отражается
  отсутствием/пометкой в `clean_posts`, но никогда — удалением raw.

#### Scenario: PK проектируется под будущее партиционирование
- WHEN создаются таблицы `raw_posts` / `clean_posts`
- THEN PK включает `time_post` (композитный `(ID, time_post)`), чтобы включение
  `PARTITION BY RANGE (time_post)` не потребовало пересборки схемы. Само
  партиционирование в этом change не включается.

## Требование 2: Очистка

Пайплайн очистки (порядок фиксирован — дешёвое раньше дорогого):

1. **Фильтр языка**: ru-only по доле кириллицы (пороги из текущей реализации),
   отбраковка с причиной `non_russian`.
2. **Junk-фильтр**: regex-категории (текущие 17) — **выносятся из кода в
   конфиг/таблицу**, чтобы пополнялись без деплоя. Отбраковка с причиной
   `junk:<категория>`.
3. **Нормализация текста**: существующая логика (разлепление склеенных токенов,
   чистка разметки) сохраняется.

#### Scenario: причина отбраковки всегда фиксируется
- WHEN документ не прошёл любой фильтр
- THEN причина пишется (в `clean_posts` со статусом либо в лог-таблицу
  отбраковки) и доступна для статистики по источнику.

---

## Требование 3: Дедупликация

Трёхступенчатая, как сейчас, но с требованиями к размещению состояния:

1. **Exact**: hash нормализованного текста, проверка уникальным индексом в БД.
2. **Near**: MinHash 128 перестановок, шинглы word-5 + char-17, `content[:4000]`,
   LSH 32 бэнда × 4 строки. **LSH-индекс живёт в памяти воркера**; БД
   используется как fallback-подтверждение между воркерами (текущая механика),
   но не как первичный поиск кандидатов на каждый документ.
3. **Confirm**: кандидаты из LSH подтверждаются levenshtein-порогом (текущие
   пороги).

- MinHash реализуется **векторизованно** (numpy, батч по всем шинглам документа),
  не per-shingle `update()`. Бенчмарк-ориентир: ≤6 мс/документ на ядро.
- Дубликат получает `is_duplicate=true`, `ID_canonical_post` → оригинал,
  `dup_score`.

#### Scenario: дубликат не эмбеддится
- WHEN документ помечен `is_duplicate=true` или отбракован фильтрами
- THEN `embedding IS NULL`, вызов эмбеддера не производится.

---

## Требование 4: Эмбеддинги

- Модель `text-embedding-3-large` с параметром `dimensions=1024`.
- Вызовы **батчами** (до лимита API), не по одному документу.
- Только для чистых недубликатов (см. сценарий выше).
- HNSW-индекс `vector_cosine_ops` на `clean_posts.embedding`; при массовой
  первичной загрузке индекс создаётся **после** заливки, не до.

#### Scenario: повторный прогон не переэмбеддит
- WHEN воркер обрабатывает документ, у которого `embedding IS NOT NULL`
- THEN вызов эмбеддера пропускается (идемпотентность = деньги).

---

## Требование 5: Производительность

Целевые числа (измерены бенчмарком, не взяты с потолка):

| метрика | сейчас | цель |
|---|---|---|
| очистка+дедуп, мс/док (1 ядро) | ~345 | ≤25 |
| пропускная способность (4 ядра) | 10.4k/час | ≥400k/час |

Обязательные механики:

1. **Батчевый claim**: `SELECT ... FOR UPDATE SKIP LOCKED LIMIT 100`, обработка
   пачки, запись результатов **одной транзакцией** (одна bulk-вставка в
   `clean_posts` + один update статусов). Per-document транзакции запрещены.
2. **Multiprocessing** по ядрам; общее состояние дедупа — через существующий
   DB fallback.
3. Профилирование до/после (`cProfile`, 100 документов) прикладывается к PR —
   чтобы подтвердить, куда ушли 323 мс.

#### Scenario: устойчивость к падению посреди батча
- WHEN воркер упал после claim, но до commit
- THEN задачи возвращаются в очередь по timeout (текущая механика lease/retry
  сохраняется), повторная обработка не создаёт дублей строк (upsert по
  `ID_raw_post`).

---

## Требование 6: Наблюдаемость и статистика источников

По каждому `ID_source` агрегируется (SQL-view или таблица, обновление по
расписанию):
`total_raw, pct_junk, pct_non_russian, pct_duplicates, avg_content_len,
last_seen_at`

Это задел под agent_memory (приоритизация источников) — сама память вне скоупа.

---

## Критерии приёмки

1. Прогон на текущем корпусе 57k raw: результат очистки/дедупа эквивалентен
   текущему (допуск: расхождение ≤0.5% по множеству отбракованных — за счёт
   порядка обработки).
2. Скорость на 4 ядрах ≥400k raw/час без эмбеддингов; с эмбеддингами упирается
   только в rate limit API эмбеддера.
3. `embedding` заполнен у 100% чистых недубликатов и NULL у 100% дублей/мусора.
4. Повторный запуск на уже обработанном корпусе: 0 LLM/эмбеддинг-вызовов,
   0 изменённых строк.
5. cProfile-отчёт до/после в PR.
6. Ни одна строка `raw_posts` не удалена и не изменена процессом очистки.

## Решённые открытые вопросы (2026-07-17)

1. Отбраковка → строка в `clean_posts` со статусом/`drop_reason` (не отдельная
   лог-таблица). Инвариант: «clean_posts = вердикт очистки по каждому raw».
2. Junk-категории → таблица в БД (пополнение без деплоя).
3. Пороги levenshtein/LSH → переносятся как есть; перекалибровка — отдельный
   change после (иначе критерий приёмки №1 теряет смысл).
